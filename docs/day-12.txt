3. TECHNICAL BREAKDOWN (INTERVIEW PREP)
---------------------------------------

Q: Why separate the Streak Engine logic from the Habit Model?
A: Separation of concerns. A Habit describes what the user wants to do, but the Streak is an aggregated performance metric over time. Keeping them separate prevents bloated models and allows us to update the streak frequently without triggering re-renders or updates on the core habit definition. It also makes gamification features (like leaderboards) easier to implement by just querying the `streaks` table.

Q: Why calculate the streak using a Server-side Edge Function instead of only on the client?
A: Security and atomicity. If we rely solely on the client, users could spoof their device time or tamper with the payload to artificially inflate their streak. A server-side Edge Function provides a single source of truth, prevents cheating, and natively handles race conditions when multiple devices sync at the same time.

Q: How does the "Local Fallback" work if the user is offline?
A: If the Edge Function fails (e.g., due to no network), `StreakService` catches the exception and computes the new streak locally based on cached data in Hive. It optimistically updates the UI so the user sees their streak go up instantly. In the background, `SyncService` queues the true completion event to be resolved by Supabase when the connection is restored.

Q: What is a "Streak Shield" and how does the data structure support it?
A: A Streak Shield allows a user to miss a day without their streak resetting to zero. In the data structure, we track `shields_held` (integer). When a "missed" action occurs, the Edge Function checks if `shields_held > 0`. If so, it decrements the shield by 1 and preserves the `currentStreak`; otherwise, the `currentStreak` is reset to 0, and the `comebackCount` goes up.
