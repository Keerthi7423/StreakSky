3. INTERVIEW PREPARATION - KEY TALKING POINTS
---------------------------------------------

Q1: Why did you use JSONB for the habit frequency?
A1: Habit frequency is non-linear. Some users want it daily, others on specific days, others just "3 times a week". Storing this as a JSONB column in Postgres (Supabase) allows us to be flexible without creating a complex schema with many nullable columns or joins.

Q2: How did you handle state management for the habit list?
A2: I used Riverpod's 'FutureProvider' to fetch the habits. This allows the UI to easily handle Loading, Data, and Error states using the .when() pattern. I used a 'StateNotifier' for the 'HabitController' to handle the side-effects like adding or deleting habits, and then used ref.invalidate() to refresh the list automatically.

Q3: How do you handle offline-first or demo scenarios?
A3: For Day 7, I implemented a 'Demo Mode' logic inside the providers. This is a great talking point for interviews because it shows you've thought about "Offline UX" or "Preview UX". If the user is in demo mode, we bypass the repository calls and return local constants, ensuring the recruiter or user always sees a beautiful, populated app.

Q4: What's the benefit of using the Repository Pattern here?
A4: It separates the domain logic (what the app does) from the data source logic (how it fetches data). If we decide to switch from Supabase to Firebase or local SQLite in the future, we only change the implementation of the repository, not the UI or the Controllers.

================================================================
TROUBLESHOOTING: MISSING COLUMN ERROR
================================================================

If you see an error: "column habits.sort_order does not exist", run the 
following SQL in your Supabase SQL Editor:

ALTER TABLE public.habits ADD COLUMN IF NOT EXISTS sort_order INTEGER DEFAULT 0;

This will add the missing column and allow you to re-enable custom 
sorting in the future.
