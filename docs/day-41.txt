Day 41: Continuous Deployment (CD) and Database Backups
=======================================================
Goal: Automate the build process for Android and iOS using GitHub Actions.

1. Android Continuous Deployment (Task 121)
-------------------------------------------
Continuous Deployment (CD) for Android automates building and uploading the app directly to the Play Store. This significantly reduces manual effort during release days.

- What we did (`.github/workflows/cd-android.yml`):
  We created a new GitHub Action that triggers automatically when we push a version tag (e.g., `v1.0.0`) or when triggered manually.
  
  The workflow does the following:
  - Sets up the Java and Flutter environment.
  - Decodes our Base64-encoded upload keystore (securely stored in GitHub Secrets) back into a file.
  - Generates the `key.properties` file securely using secrets so Flutter can sign the release build.
  - Runs `flutter build appbundle --release` to generate the `.aab` file.
  - Uploads the `.aab` file directly to the Google Play Store's Internal Testing track using the `r0adkll/upload-google-play` action.

  *Interview Tip*: Be prepared to explain the difference between an APK and an AAB (Android App Bundle). Google requires AABs for new apps because they allow the Play Store to deliver customized, smaller APKs tailored to each user's device architecture, screen density, and language. Also, explain that we store our sensitive keystore as a Base64 encoded string in GitHub Secrets because Git repositories should never contain binary signing keys directly.

2. iOS Continuous Deployment (Task 122)
---------------------------------------
Deploying to iOS requires a bit more strict certificate and provisioning profile handling than Android, but we can still automate it.

- What we did (`.github/workflows/cd-ios.yml`):
  We built a workflow specifically running on macOS (`macos-latest` runner).
  
  The workflow does the following:
  - Sets up the Flutter environment on a Mac runner.
  - Runs `flutter build ipa --release` to compile the app for iOS.
  - Uses the App Store Connect API keys (stored in GitHub Secrets) to authenticate with Apple.
  - Uploads the `.ipa` directly to TestFlight using `apple-actions/upload-testflight-build`.

  *Interview Tip*: Discuss how the transition to App Store Connect API keys simplifies CI/CD. In the past, developers had to install specific certificates and provisioning profiles on CI runners manually, which often failed. With the API key (`Issuer ID`, `Key ID`, and `.p8` Private Key), Apple allows the CI runner to communicate and publish without needing a keychain locally.
