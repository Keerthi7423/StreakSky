----------------------------------------------------------------
3. INTERVIEW PREPARATION (Q&A)
----------------------------------------------------------------

Q1: Why use a separate 'streaks' table instead of just counting completions?
A: Performance and Query Complexity. Counting thousands of completions across 
   hundreds of habits every time the app opens is expensive. Pre-calculating 
   and storing the current streak allows for O(1) retrieval, ensuring a 
   snappy user experience.

Q2: How do you handle "Optimistic UI" with streaks?
A: When a user taps 'complete', we immediately update the local state 
   (HabitCard) and trigger the background update. If the network fails, 
   the sync service handles the retry. We invalidate the provider only 
   after the logic completes to ensure the UI stays in sync with truth.

Q3: Explain the logic behind the Streak Shield cap.
A: The cap of 3 prevents users from "gaming the system." It strikes a 
   balance between providing a safety net and maintaining the integrity 
   of the streak mechanic. It encourages consistency while being forgiving.

Q4: How does the family provider work in Riverpod?
A: `FutureProvider.family<Result, Param>` allows us to pass a unique 
   parameter (like `habitId`) to a provider. Riverpod automatically 
   caches the result for that specific ID, so multiple widgets watching 
   the same habit don't trigger multiple network calls.
